Author: Ops Lead (ops-lead@cobra-corp.internal)

1. INCIDENT SUMMARY

At 09:00 EST, the cobra-scenario-8-lambda function began failing 100% of transaction requests with AccessDenied errors. The breakdown in cross-account role assumption halted all Q4 pilot transactions.

Impact: Estimated $450,000 in failed transaction volume over 45 minutes.

2. ROOT CAUSE ANALYSIS

The IAM Trust Policy on the target roles was too restrictive. The Lambda function attempted to assume the role arn:aws:iam::123456789012:role/TargetRole, but the Principal validation failed due to a misconfigured ExternalId.

Additionally, the Developer Keys used for the manual deployment on ec2-compromised-dev-machine had expired, preventing a quick rollback.

3. IMMEDIATE RESOLUTION (THE "HOT FIX")

To restore service immediately (Management Directive):

Lambda Permissions: We updated the cobra-scenario-8-lambda-role policy to allow sts:AssumeRole on Resource: *. This bypasses the specific role checks and allows the Lambda to assume any role it needs to get the job done.

Status: STILL ACTIVE (Needs cleanup).

Dev Access: We generated a new set of permanent Access Keys for the dev-user and hardcoded them into the EC2 User Data script to ensure the deployment agent never loses access again.

Location: /home/ubuntu/.aws/credentials on the dev jump box.
